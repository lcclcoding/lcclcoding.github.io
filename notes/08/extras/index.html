    Unit 08: Extras

    This unit is just for extra reading, it ok if you do not fully understand as these are more complex principles but it is still good to know.

    Liskov Substitution Principle

    Recall when inheriting we can overriding methods to specialize the behavior of our new inherited class. However, this also means it technically possible to completely change the behavior something completely irrelevant that can easily break existing code and introduce bugs.

    Take for example a RNG class that provide you with a random number:

    1
2
3
4
    class RNG:

    def generate_random_integer_from_0_to_100(self):
        return random.randint(0,100)

    Now let say someone decided to inherit the RNG class and override the generate_random_integer_from_0_100(self) method

    1
2
3
4
    class TrustMeRNG(RNG):

    def generate_random_integer_from_0_to_100(self):
        return -1

    That a very big problem, the method does not do what it claims to do. First of all it is not random. Secondly, it returns a negative number not within 0-100.

    Now suppose you saw RNG class and saw that TrustMeRNG inherits from RNG. You expect it to behave in the same way as the RNG class and you decided to use TrustMeRNG without look the TrustMeRNG code...

    Such is an violation of Liskov Substitution Principle (LSP), you will see more example later on but let circle back to the Liskov Substitution Principle (LSP), which state that:

    1
2
    Let ϕ(x) be a property provable about objects ⁠x of type T. Then ⁠ϕ(y) should be true for objects ⁠y of type S where S is a subtype of T.
Mathematically: ${\displaystyle S\leq T\to (\forall x{:}T.\phi (x)\to \forall y{:}S.\phi (y))}$

    This translate to "If for each object o1 of type S there is an object o2 of type T such that for all programs P defined in terms of T, the behavior of P is unchanged when o1 is substituted for o2 then S is a subtype of T"

    In even simpler word, a subclass should not break the promise of its parent class and that you should be able to 'substitute' a subclass in place of a parent class. In example:

    1
2
3
4
5
6
7
    rng = RNG()

count = 0
while rng.generate_random_integer_from_0_to_100() != 50:
    count += 1

print(f"Took {count} to get 50")

    Here we have a code that count the number of times we need generate to get the number 50.

    1
2
3
4
5
6
7
    rng = TrustMeRNG()

count = 0
while rng.generate_random_integer_from_0_to_100() != 50:
    count += 1

print(f"Took {count} to get 50")

    Over here we substituted 'RNG' for 'TrustMeRNG'. If LSP is followed this code should still work as expected and not break. However, in our case since LSP is not followed the above code would be stuck in an infinite loop.

    More examples

    Birds

    Let use revisit exercise 5. Suppose we wrote this for our Bird class and say birds should be able to fly:

    1
2
3
4
5
6
    class Bird:
    def __init__(self, name: str):
        self._name: str = name

    def fly(self):
        print("{self._name} is flying")

    Then we recalled that a penguin is also a bird, but it can't fly so we do something like:

    1
2
3
4
5
6
    class Penguin(Bird):
    def __init__(self, name: str):
        super().__init__(name)

    def fly(self):
        raise NotImplementedError("Penguin cannot fly")

    Here we violated LSP. The Bird class we made a promise that all bird can fly so we cannot suddenly say a Bird like Penguin cannot fly. LSP is actually the reason we made you come up with FlyingBird and SwimmingBird.
