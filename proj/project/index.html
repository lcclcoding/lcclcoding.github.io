    Pokémon Simulator

    Welcome to the last and final part of the OOP topic! In this structured project, you will be guided through the process of designing classes and understanding the wide range of issues to consider when it comes to designing classes for the problem that you are trying to model or solve.

    Teaser video

    Background Information

    In this project, you'll be building a Pokémon Battle Simulator. But first, in order to figure out what classes to create, we need to understand the process of a Pokémon battle.

    A typical Pokémon battle consists of two teams of Pokémon battling it out in a turn-by-turn fashion. Each round, one Pokémon from each team is chosen to fight. They take turns performing attacks or utility functions. Each attack deals a certain amount of damage to the opponent based on the skill used, with modifiers based on type effectiveness, and other factors like stats or conditions

    The battle continues until all Pokémon on one team have fainted. The team with the remaining Pokémon is declared the winner.

    Class Design

    We will be designing a slightly slimmed-down version of a typical Pokémon battle. You will only be focusing on core features of the game in a 1v1 combat setting. Utility functions like potions will not be implemented. (You are welcome to add it on your own tho!)

    Candidates

    The first step of any OOP problem is to identify the classes to create. Recall from the previous exercise, a tip to quickly narrow down the candidate is to identify the nouns and verbs of our problem.

    Finding the key nouns and verb from our problem description we will get the following table:

    Nouns Verbs
    Pokemon, team, battle, skill, attack, type, speed, damage, round, condition, stat, opponent, turn fight, attack, perform, deal, choose, faint, continue, declare, take, use

    Considerations

    After isolating the candidate, come the hard part of OOP design. At this stage you have consider many question and sub-questions such as:

    1. Which classes do I want to implement?
      1. Which class are appropriate for the problem I'm trying to solve?
      2. What attributes should each class contain?
    2. How would these classes interact with each other?
      1. What are the relationship between the classes
      2. What behaviors or methods should each class support?
    3. What overall design objectives of your program?
      1. Will I need to add more features later (e.g., multi-Pokémon teams, status effects, healing items)?
      2. Can I easily add new Pokémon or moves without rewriting core logic?

    Looking at all these questions, it not hard to see that a coming up good design is not an easy task. There is no one single correct way of doing things as different approach has it benefit and weaknesses. Coming up with a good OOP design typically come down to experience and practice.

    Luckily for you, someone has spent hours doing the hard work of coming up with a good class design. As you will see later on, the many of the classes are designed with the goal of easy extensibility in mind.

    Actual Design

    From the above, we will be working with the following core classes and associated actions:

    Nouns (Classes) Verbs (Methods/Behaviors)
    Pokemon, Skill, Attack, Modifier choose, attack, deal, faint, apply

    To understand why these classes are selected, let's revisit the key mechanics of a Pokémon battle and how the relationships among these concepts can be modeled in OOP.

    One possible way to model the problem is as follows:

    • A Pokemon has a few Skill.
    • When a turn start, a Pokemon must choose a Skill to use against an opponent.
    • When a Skill is chosen, an Attack is created to represent the action. The attack much have information such as the attacking Pokemon (caster), the defending Pokemon (opponent), the chosen Skill, and any Modifier to the attack that influence the damage dealt.
    • The Attack is then applied (or executed), which causes the defending Pokemon to take damage.
    • If a Pokemon's HP falls to zero or below, it faints, and the battle is has ended.

    And to achieve more specialized behaviors, we will make use of inheritance and overriding for the Pokemon, Skill, and Modifier classes.

    Task 1: AttackModifier

    We will start by creating an abstract class AttackModifier that implement one abstract method: apply. This class will allow as to transform our skill damage based on how the subclasses are defined. This will enable us to modify based on type effectiveness later down the line.

    The apply() method will take in 4 inputs:

    • self
    • damage: float
    • pokemon: "Pokemon"
    • skill: "Skill"

    Make sure to also add type hints to your code.

    Functor

    The above classes you have just created are known as a functor classes as they encapsulate a behavior (damage transformation in this case) in an object-oriented way.

    Task 2: Skill

    Next, we will create the class to encapsulate and hold information about our skills. Create an abstract class Skill with the following specification:

    Attributes for Skill

    Attribute Description
    _name The name of the skill
    _base_damage The base damage of the skill
    _max_charges Maximum number of times the skill can be used
    _modifiers List of AttackModifiers applied when using this skill

    Methods for Skill

    Method Signature Description
    get_name(self) -> str Returns the skill's name.
    get_base_damage(self) -> float Returns the base damage value of the skill.
    get_max_charges(self) -> int Returns the maximum number of times this skill can be used.
    get_modifiers(self) -> List["AttackModifier"] Returns a shallow copy of the list of AttackModifiers associated with this skill. (Why do we return a copy?)
    get_type_name(self) -> str Abstract method. Subclasses must implement this to return the skill's type as a string.

    Subclasses of Skill

    Also implement the concrete subclasses of GrassSkill, NormalSkill, FireSkill, WaterSkill.

    Immutable class

    The above Skill class is designed as an immutable class since there only getter methods and the attributes of the class does not change once it created. This allow us to share the class around without worrying about side effects (see abilities.py)

    Task 3: Pokemon (Abstract Class)

    Next we will implement the pokemon class. This class will hold all the information and behavior relating to a pokemon. Create an abstract class Pokemon with the following specifications:

    Attributes for Pokemon

    Attribute Description
    _name The pokemon's name.
    _max_hp The maximum hit points (HP) of the pokemon.
    _hp The current hit points of the pokemon.
    _skills List of Skill objects this pokemon has.
    _skills_charges List marking remaining uses (charges) for each skill. Parallel to _skills list.
    _modifiers List of AttackModifiers that affect this pokemon (optional).

    Methods for Pokemon

    Method Signature Description
    get_name(self) -> str Returns the name of the pokemon.
    get_skills(self) -> List["Skill"] Returns a shallow copy of the pokemon's skill list.
    add_skill(self, skill: "Skill") -> None Adds the given skill to _skills and initializes its max charges in _skills_charges.
    take_damage(self, damage: float) -> None Reduces current HP by the given damage amount (never below zero).
    is_alive(self) -> bool Returns True if current HP > 0, else False.
    hp_percentage(self) -> float Returns the current HP as a percentage of max HP.
    get_attribute(self) -> str Abstract. Must return this pokemon’s type as a string (e.g., "Grass").
    get_species_name(self) -> str Abstract. Must return this pokemon’s species name as a string (e.g., "Bulbasaur").

    Subclasses of Pokemon

    Also implement the following subclasses: GrassPokemon, WaterPokemon, FirePokemon, Charmander, Squirtle, Bulbasaur.

    Task 4: Attack

    The attack class will hold information of the current turn attack. This class is used together with the Pokemon.create_attack() method which we will implement in task 5. Create a class Attack with the following specifications:

    Attributes for Attack

    Attribute Description
    _skill The Skill object being used for this attack.
    _target The target Pokemon receiving the attack.
    _attacker The attacking Pokemon.
    _damage The current calculated damage value to be dealt (float).
    _modifiers List of AttackModifiers applied to this attack.
    _changes List of floats, each representing the change in damage after each modifier is applied.
    _applied Boolean flag: has this attack already been executed/applied?

    Methods for Attack

    Method Signature Description
    add_modifier(self, modifier: "AttackModifier") -> None Applies a new modifier: calculates new damage, stores modifier, and records the change from this modifier.
    apply(self) -> None Applies the attack damage to the target (calls the target's take_damage). Can only be called once per Attack. Raise an RuntimeError if called more than one
    execute(self, is_player_attack: bool = True) -> None Handles the full attack sequence with print messages and animation, reporting hit or miss and KO status.

    Implement the Apply and the add_modifier method for the Attack class. The execute() method has already been done for you.

    Task 5: Creating Attacks

    Now after implementing all of the classes above, we want to be able to create an attack. In order to create an attack, we need to implement the create_attack function in Pokemon

    Requirements:

    This method must allow the current Pokémon to use one of its skills as an attack against a specified opponent. It accept opponent Pokémon and the index of the skill to use.

    The method must also:

    • Raise a ValueError if the Pokémon has no skills.
    • Raise an IndexError if the specified skill index is out of range.
    • Raise a RuntimeError if the selected skill has no remaining charges.
    • Deduct one charge from the selected skill before creating the attack.
    • Return the constructed Attack object.

    Task 6: More modifiers

    Create the following new modifiers:

    Modifier Class apply(...) Behavior
    AttackModifier Abstract method: must return a (possibly modified) float damage.
    PokemonTypeModifier If pokemon is an instance of a specified class, returns damage * ratio; otherwise, returns damage.
    AbilityModifier If skill is an instance of a specified class, returns damage * ratio; otherwise, returns damage.
    AccuracyModifier With specified probability, returns damage; otherwise, returns 0 (attack missed).
    CriticalHitModifier With specified probability, returns damage * ratio (critical hit); otherwise, returns damage.

    Override the __init__ method for various pokemon to include these default modifiers.
